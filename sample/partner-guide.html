1. はじめに

1.1 このガイドの目的

このガイドは、eBay APIをご利用のパートナー様がTrading API等の従来API（SOAP/XML）から、Modern APIs（REST/GraphQL）への移行を円滑に進めるための実践的な手引きです。

対象読者: eBay APIを使用している技術担当者・開発者

1.2 Pilotプログラムのゴール

本Pilotプログラムでは、以下の3つを目標としています：

  1. 現在使用しているAPIの棚卸し
    • どのTrading API呼び出しを使っているかのリストアップ
  2. 新APIでの実現方法の検証
    • 各APIの移行先（REST/GraphQL）の確認
    • サンプルコードでの動作確認
  3. 自社システムへの影響の洗い出しと、eBayへのフィードバック
    • 技術的な制約・課題の把握
    • 移行工数の見積もり
    • 必要なご要望・改善リクエストのeBayへの提出

1.3 Pilot実施期間（目安）

Pilotの実施期間は 2026年6月末ごろ を目安としています。これは「この日で使えなくなる」という締切ではありません。6月末を過ぎても、Modern APIs やツール（Skills / MCP）は引き続きご利用いただけますし、困ったときのご相談もこれまで通りお受けします。

この目安期間は、早期のフィードバックを特に歓迎している期間です。早めに触って、自社への影響やご要望を早いうちにお寄せいただくのがおすすめです（くわしくは 2.1 Pilotプログラムの意義 を参照）。

2. 移行の全体像

まず全体像：移行は「3つの区分」で考える

eBay APIは「従来 vs Modern」の2分ではなく、実務上は次の3区分に分かれます。自分がどれを使っているかで、やることが変わります。

判断基準はシンプル：SOAP/XML ベースのもの（Trading API 等の従来API・Auth'n'Auth・Platform Notifications）は、すべて移行対象です。REST/GraphQL（＝Modern）は移行"先"で、すでに使っていれば対応不要。

区分代表的なAPIあなたがすべきこと
① Legacy
従来		 Trading API / Post-Order API / Platform Notifications / Auth'n'Auth（SOAP・XML） 移行が必要。本ガイドの対象はこれ。
② 既存Modern
一般提供済み（GA）		 Taxonomy / Metadata / Fulfillment / Finances / Identity / Browse / Catalog※ すでに使っていれば対応不要。移行の"先"であって"元"ではない。
③ New（新API）
本Pilotの主役		 Listing / Order / Item Commitment / Order Inquiry / Return / Cancellation / Offer / User …（GraphQL）＋ Inventory Mapping 新しく提供されるもの。Legacyの主要機能（特に出品・注文）の移行先。

「New（③）」は「Modern」の一部です：新API（③）はすべて Modern に含まれますが、逆に Modern がすべて新しいわけではなく、既にGAの既存API（②）も含みます。「Modern＝全部これから」ではありません。例：Taxonomy API を既にお使いなら、その機能は②なので移行は不要です。

※Catalog API は限定公開（要アクセス申請）。

※Inventory Mapping は既存在庫を移行するツールではなく、外部商品データから AI が出品プレビューを生成する支援API（US限定・1回最大10件）。生成結果を Listing API での出品作成に利用します。

2.1 なぜ今、移行が必要なのか？

eBayは2026年12月を目標に、従来のTrading API（SOAP/XML）からModern APIs（REST/GraphQL）への移行を推進しています。

移行の背景:

  • 技術的な刷新: 20年以上前の技術スタック（SOAP/XML）から、現代的なREST/GraphQLへ
  • パフォーマンス向上: 必要なデータのみを取得できるGraphQL、軽量なJSON形式
  • 開発者体験の改善: 直感的なAPI設計、豊富なドキュメント、SDK対応
  • セキュリティ強化: OAuth 2.0による標準的な認証・認可

Pilotプログラムの意義:

本Pilotの目的は、Modern APIsの仕様が固まる前の段階で、実際にAPIを使う皆様から先行してご意見・ご要望（フィードバック）を集めることです。移行そのものをこの期間内に完了していただくためのものではありません。

早期にご参加いただくメリット:

  • 仕様が固まる前に、自社のツール・システムへの影響を洗い出せる。本番移行の前に、準備と工数見積もりを進められます。
  • 気づいた点・ご要望を早い段階で共有できる。いただいたフィードバックは、今後の改善検討の参考にさせていただきます。

そのため、まず実際にAPIを触って自社への影響点を洗い出し、気づいた点は早めにお寄せいただくことをおすすめします。なお、いただいたフィードバックの反映を確約するものではありません。

データ形式 XML JSON パース処理の変更

2.3 機能別の変更マップ（どの機能がどう変わるか）

以下は、パートナー様がよく使用する機能の移行先をまとめたものです。

一目でわかる：ドメイン別 移行マトリクス

大原則は「SOAP/XML ＝ 移行対象」。下表で、自社が使うドメインの移行先と対応関係（1:1 / 1:N / 対応なし）をまず確認してください。各ドメインの掘り下げは 2.3.1 以降、コール単位の早見表は 4.2.2、正確な置換は Skills で確認してください。

ドメイン従来API（SOAP/XML）移行先（Modern）対応関係区分
認証 Auth'n'Auth（FetchToken 等） OAuth 2.0（＋ Identity API） 全面置換（一部コールは不要／未定） 必須・基盤
出品 AddItem / AddFixedPriceItem / ReviseItem / RelistItem / EndItem / GetItem / GetSellerList Listing API（GraphQL） N:1（多数のコール → 1 API の各 mutation/query） ③ New
在庫 ReviseInventoryStatus Listing API bulkUpdateInventory（GraphQL） 1:1 ③ New
注文取得 GetOrders Order API（GraphQL）＋ Fulfillment API（REST）／ Item Commitment API（GraphQL） 1:N（支払済／未払で分割） ③＋②
注文処理 CompleteSale markItemCommitmentsPaid / markOrdersPaid / createShippingFulfillment / leaveFeedback / provideShipmentInfoForItemNotReceivedInquiry 1:5（用途別に分割） ③＋②
商品情報・カテゴリ GetCategories / GeteBayDetails / GetCategoryFeatures / GetCategorySpecifics Taxonomy API ＋ Metadata API（＋ Catalog API） N:N（用途別メソッドへ） ② 既存Modern
ユーザー・アカウント GetUser / GetAccount User API（GraphQL）／ Identity API（REST）／ Finances API（REST） 1:N ③＋②
メッセージ・評価 GetMemberMessages / GetFeedback Message API（REST）／ Feedback API（REST） 評価は 1:2（エントリ／サマリ） ② 既存Modern

※メッセージ／評価の Message API・Feedback API は 2025Q4 提供開始の新しい REST（Message API は Limited Release＝要アクセス申請）。Trading から使っている場合は移行が必要で「対応不要」には該当しません。最新の提供状況・スコープは Skills/MCP で確認してください。

2.3.1 認証（→ 実装手順は §4.3）

Auth'n'Auth → OAuth 2.0。長期トークン（約18ヶ月）から Access Token（2時間）＋ Refresh Token 方式へ変わり、トークン更新ロジックの新規実装が必須です。FetchToken は不要（OAuthフローで取得）、GetTokenStatus / RevokeToken は置換未定（"more details to follow"）。スコープの明示指定が必要になります。

2.3.2 出品管理 — データモデルが根本から変わる（最重要）

APIは createListing の1ステップですが、「商品とは何か」の捉え方が変わります。旧API（AddItem）は「Item」が中心。新API（Listing API）は「商品（product）」と「販売単位（items＝SKU）」を概念として分離します：

createListing（出品作成）
├── product  ＝ 共通の商品情報（何を売るか）
│             例: title / description / aspects（Item Specifics）/ images / category
└── items    ＝ 販売単位＝SKU（どう売るか）
              例: price / quantity / 商品識別子（GTIN・ePID）/ traits（バリエーション）

※上図は概念図です。実際のフィールド名・階層（listingItems.items 等）や各フィールドの正確な位置は、Skills / MCP で最新スキーマを確認してください。

このため既存システムのデータモデル・クラス設計の見直しが必要になる場合があります。あわせて createListing は1コール1出品（一括は Sell Feed API）、updateListing の配列（画像・aspects 等）は差分でなく全置換（部分更新の感覚で事故りやすい）。

2.3.3 注文管理 — 1コールが用途別に分かれる

GetOrders は支払状況で参照先が分かれます：支払済 = Order API（GraphQL）＋ Fulfillment API（REST）／未払 = Item Commitment API（GraphQL）。

CompleteSale は用途別に5つへ分割。自社がどの用途で使っているかを洗い出して振り分けます：

  1. markItemCommitmentsPaid — 未払い注文を支払い済みに（Item Commitment API）
  2. markOrdersPaid — オフライン注文を支払い済みに（Order API）
  3. createShippingFulfillment — 配送情報入力（Fulfillment API）
  4. leaveFeedback — フィードバック入力（Feedback API）
  5. provideShipmentInfoForItemNotReceivedInquiry — INR inquiry 用配送情報（Order Inquiry API）

EU/UK 向けの金銭関連操作（返金・キャンセル・Finances 等）はメッセージのデジタル署名が必須（越境セラーは要対応）。

2.3.4 商品情報・カテゴリ — ほぼ「② 既存Modern」で対応不要

カテゴリ系はすべて公開済みの既存Modernに対応します：GetCategories（ツリー）→ Taxonomy API（getCategoryTree）、GetCategorySpecifics（Item Specifics）→ Taxonomy（getItemAspectsForCategory）、GetCategoryFeatures（出品条件）→ Metadata API、GeteBayDetails（配送・通貨等のサイト情報。※カテゴリツリーは含まない）→ Metadata。Product Catalog 検索は Catalog API（限定公開）。

→ すでに Taxonomy 等をお使いなら、この領域は移行済みで追加作業は不要です。

2.3.5 ユーザー・アカウント情報

GetUser → User API（GraphQL, 詳細） と Identity API（REST, 基本情報） の使い分け。GetAccount → Finances API（getTransactions / getBillingActivities）。取得範囲が自アカウントのみになる点は要確認。なお GetAccount／Finances 系（および返金・キャンセル等の金銭操作）は、EU/UK 居住セラーではメッセージのデジタル署名が必須（§2.3.3 参照）。

2.3.6 メッセージング・フィードバック

GetMemberMessages → Message API、GetFeedback → Feedback API（エントリ取得とサマリ取得の2つに分割）。いずれも 2025Q4 提供開始の新しい REST APIで、Message API は Limited Release（要アクセス申請）。Trading のメッセージ／評価を使っている場合は移行が必要です（"既に使っていれば不要" には該当しないことが多い）。最新の提供状況・スコープは Skills/MCP で確認してください。

2.4 移行の進め方（推奨順）

各機能の影響度・対応関係は 2.3 のマスター表を参照。着手順は、全APIの前提となる認証から始め、影響の大きい順に進めます：

  1. Week 1-2 ｜ 認証（OAuth 2.0） — 全API共通の前提。最優先（影響度 ★★★）。
  2. Week 3-4 ｜ 出品（Listing API） — データモデル変更を伴い影響最大（★★★）。
  3. Week 5-6 ｜ 注文・その他 — CompleteSale の用途洗い出しと振り分け（★★〜★★★）。

影響度 ★（カテゴリ・ユーザー・メッセージ）は後回しでも可。とくに「② 既存Modern」を既にお使いの領域は対応不要です。

2.5 よくある質問（移行前の不安）

Q: 全てのAPIを一度に移行する必要がありますか？

A: いいえ。段階的な移行が可能です。

  • 従来APIとModern APIsは並行稼働できます
  • まず認証（OAuth 2.0）を実装
  • その後、機能ごとに段階的に移行

Q: 既存の出品データはどうなりますか？

A: ライブ出品そのものは保持されます（eBay上の出品レコードは、呼び出すAPIを変えても消えず、再出品も不要）。ただし「まったく影響を受けない」わけではありません。

  • 再登録的なデータ移行は不要。既存の出品は ID（listingId）で新しい Listing API から取得・更新できます（旧 Inventory API のような変換ステップは不要）。
  • ただし"扱い方（データモデル）"は変わります：取得・更新は新モデル（product / aspects / items / traits）経由になり、Trading の XML 構造とは異なります。連携コードの読み書き処理は改修が必要です。
  • updateListing の配列（画像・aspects 等）は全置換。また一部の従来フィールドは移行対象外（新モデルに対応物がない）。
  • → まとめ：出品データは残り移行も不要だが、アクセスの仕方（モデル）は新形式に変わる。個別の出品・項目が新モデルでどう扱われるかは Skills で確認してください。

Q: 移行にどれくらいの工数がかかりますか？

A: 実装規模によりますが、目安は以下の通りです：

実装規模 推定工数
小規模（基本的な出品・注文管理のみ） 2-4週間
中規模（出品・注文・在庫管理） 1-2ヶ月
大規模（全機能統合） 2-3ヶ月

※Pilot期間中に正確な見積もりを出すことが目標です。

Q: 移行しないとどうなりますか？

A: 2026年12月以降、従来APIが廃止される可能性があります。

  • 具体的な廃止日は未定ですが、早期移行を推奨
  • Pilot参加により、移行計画を立てる時間的余裕が得られます

3. AI支援ツールのセットアップ

eBayでは、パートナー様の移行を支援するためのAI支援ツールを2つ提供しています：

  1. Public API Migration Skills - AIアシスタントに移行知識を統合
  2. eBay Public API MCP Server - AIアシスタントから直接eBay APIを呼び出し

3.1 前提条件

以下の環境を事前にご準備ください：

  • Node.js 22以上（MCP Serverの要件。Skillsのみの利用なら14以上で可） 
    node --version  # v22 以上であることを確認
  • AIアシスタント（以下のいずれか）
    • Claude Code ※推奨
    • Cursor
    • GitHub Copilot
  • eBay Developer Account
    • アカウント作成: developer.ebay.com

3.2 Tool 1: Public API Migration Skills のインストール

3.2.1 Skillsとは？

AI Skills（スキル）は、AIアシスタントに特定の専門知識を持たせるための仕組みです。 eBayが提供するSkillsをインストールすることで、AIアシスタントがTrading API → Modern APIsの移行方法を即座に回答できるようになります。

3.2.2 提供されるSkills

以下の9つのドメイン別Skillsが提供されます：

Skill カバー範囲
account-management ユーザープロファイル、セラーアカウント、eBay Stores設定、税設定、配送割引ルール
authorization 認証・認可フロー（Auth'n'Auth → OAuth 2.0）
communication バイヤー・セラー間メッセージング、フィードバック
listing-management 出品ライフサイクル（作成、更新、再出品、終了、検索）、画像アップロード
listing-metadata カテゴリ検索、Item Specifics、Product Catalog
notification Platform notifications、セラーアラート、イベント購読
offers-and-leads Best Offer交渉、Classified Ad leads取得
order-management 注文取得、配送fulfillment、返品、キャンセル、ケース管理
other-apis VeRO（知的財産権）レポート

3.2.3 インストール手順

Step 1: zipファイルの展開

eBay Japan Tech Supportから配布された public-api-migration-skills-1.4.0.zip を展開します。

unzip public-api-migration-skills-1.4.0.zip
cd public-api-migration-skills-1.4.0

Step 2: Skillsのインストール

npx skills add ./

全てのSkillsがインストールされます。

特定のSkillsのみインストールする場合:

# 出品管理のみ
npx skills add ./ --skill listing-management

# 複数指定
npx skills add ./ --skill listing-management --skill order-management

Step 3: インストール確認

npx skills list

ebay/public-api-migration-skills が表示されればOKです。

Step 4: AIアシスタントの再起動

Claude Code等のAIアシスタントを再起動します。

3.2.4 動作確認

AIアシスタントで以下のように質問してみてください：

How do I migrate the GetItem API to modern eBay APIs?

Skillsが正しくインストールされていれば、Listing API (GraphQL)への移行方法が返答されます。

3.3 Tool 2: eBay Public API MCP Server のセットアップ

3.3.1 MCP Serverとは？

MCP（Model Context Protocol）Serverは、AIアシスタントが外部のAPIやツールを直接呼び出せるようにする仕組みです。

eBayが提供するMCP Serverをインストールすることで、AIアシスタントからeBay APIを直接呼び出して検証できるようになります。

3.3.2 重要な制限事項

⚠️ Production環境では READ ONLY のみ

  • REST API: GETリクエストのみ（読み取り専用）
  • GraphQL API: Queryのみ（Mutationは不可）
  • POST/PUT/DELETE: 実行不可

Sandbox環境は現在サポートされていません。

3.3.3 インストール手順

Step 1: zipファイルの展開とビルド

unzip npm-public-api-mcp-1.0.0.zip
cd npm-public-api-mcp-1.0.0   # 展開されたフォルダ名に合わせて移動してください

npm install
npm run build

Step 2: eBay Developer Accountで認証情報を取得

2-1. Client ID & Client Secret の取得
  1. developer.ebay.com/my/keys にアクセス
  2. Application Keysの Client ID と Client Secret をコピー
2-2. Refresh Token の取得（User Tokenモードの場合）

User Tokenモード: プライベートデータ（自社の注文情報等）にアクセスする場合

Refresh Tokenの取得方法は OAuth Authorization Code Grant を参照してください。

Application Tokenモード: パブリックデータ（カテゴリ情報等）のみアクセスする場合はRefresh Token不要です。

2-3. 必要なOAuth Scopes

アクセスしたいAPIに応じて、適切なスコープを選択します。

参考: OAuth Scopes Documentation

よく使われるスコープ:

  • https://api.ebay.com/oauth/api_scope - 基本的な読み取り
  • https://api.ebay.com/oauth/api_scope/sell.listing.read - 出品情報の読み取り（Listing API / GraphQL。自社出品の取得は User Token が必要）
  • https://api.ebay.com/oauth/api_scope/sell.fulfillment.readonly - 注文情報の読み取り

4. 実践ガイド（Step by Step）

4.1 Step 1: 現在使用中のAPIの棚卸し

4.1.1 目的

現在のシステムで使用しているTrading APIの呼び出しを全て洗い出します。

4.1.2 実施方法

方法A: コードベースから自動抽出（推奨）

以下のようなスクリプトでAPIコールを抽出します：

import re
import os

# Trading API呼び出しパターン
api_pattern = r'(GetItem|AddItem|ReviseItem|GetOrders|CompleteSale|GetUser|SetNotificationPreferences|FetchToken|GeteBayDetails|AddFixedPriceItem|RelistItem|EndItem|GetSellerList|GetMyeBaySelling)'

def find_api_calls(directory):
    results = {}

    for root, dirs, files in os.walk(directory):
        for file in files:
            if file.endswith(('.java', '.php', '.py', '.js', '.cs')):
                filepath = os.path.join(root, file)
                with open(filepath, 'r', encoding='utf-8', errors='ignore') as f:
                    content = f.read()
                    matches = re.findall(api_pattern, content)
                    if matches:
                        results[filepath] = list(set(matches))

    return results

# 使い方
api_calls = find_api_calls('/path/to/your/codebase')

for filepath, calls in api_calls.items():
    print(f"{filepath}:")
    for call in calls:
        print(f"  - {call}")

方法B: 手動でリストアップ

以下のフォーマットでExcelやスプレッドシートに記録します：

API名 使用箇所 呼び出し頻度 重要度 備考
GetItem 商品詳細ページ 高 高
GetOrders 注文管理画面 高 高
AddItem 出品フォーム 中 高

4.1.3 優先順位付け

抽出したAPIを以下の基準でランク付けします：

  • A（最優先）: 呼び出し頻度が高く、ビジネスクリティカル
  • B（中優先）: 定期的に使用するが、代替手段がある
  • C（低優先）: 使用頻度が低い、または廃止予定の機能

Pilot期間中はAランクのAPIの移行先確認を優先します。

4.3 Step 3: 認証フローの移行

4.3.1 Auth'n'Auth → OAuth 2.0 の変更点

項目 Auth'n'Auth OAuth 2.0
トークン取得 FetchToken API Authorization Code Grant
トークン有効期限 約18ヶ月 Access Token: 2時間
Refresh Token: 18ヶ月
リフレッシュ なし Refresh Tokenで自動更新
権限管理 なし スコープベース

4.3.2 OAuth 2.0 実装の流れ

Step 1: Authorization Code の取得

ユーザーをeBay consent URLにリダイレクトします：

https://auth.ebay.com/oauth2/authorize
  ?client_id=YOUR_CLIENT_ID
  &response_type=code
  &redirect_uri=YOUR_REDIRECT_URI
  &scope=https://api.ebay.com/oauth/api_scope/sell.listing

※ redirect_uri には実際には eBay の RuName（Redirect URL name）を指定します（生のURL文字列ではありません）。

Step 2: Authorization Code の交換（Access Token と Refresh Token を同時に取得）

curl -X POST https://api.ebay.com/identity/v1/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -u "YOUR_CLIENT_ID:YOUR_CLIENT_SECRET" \
  -d "grant_type=authorization_code" \
  -d "code=YOUR_AUTHORIZATION_CODE" \
  -d "redirect_uri=YOUR_REDIRECT_URI"

Step 3: 以後の更新（Access Token が切れたら Refresh Token で再取得・2時間ごと）

curl -X POST https://api.ebay.com/identity/v1/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -u "YOUR_CLIENT_ID:YOUR_CLIENT_SECRET" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=YOUR_REFRESH_TOKEN"

Step 4: Access Token でAPIを呼び出し

curl -X GET "https://api.ebay.com/sell/fulfillment/v1/order?limit=1" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"

4.3.3 スコープの選び方

アクセスしたいAPIに応じて、適切なスコープを選択します。

注意：出品（Listing API / GraphQL）のスコープは sell.listing 系です。旧REST Inventory APIの sell.inventory 系では新 Listing API は 403 になります（別スタックのため）。

例: 出品管理（Listing API）に必要なスコープ

https://api.ebay.com/oauth/api_scope/sell.listing       # 出品の作成・更新・終了（書き込み）
https://api.ebay.com/oauth/api_scope/sell.listing.read  # 出品の取得・検索（読み取り）

詳細: OAuth Scopes Documentation

4.4 Step 4: サンプルコードでの検証

4.4.1 推奨アプローチ

  1. Aランク（最優先）のAPI 1つから開始
    • 例: GetItem → Listing API の sellerListings（GraphQL）
  2. MCPまたはPostman等でAPI動作を確認
    • Claude Code + MCP を使えば、自然言語でAPIを呼び出せます（本番は読み取り系のみ）
    • 例: List my active eBay seller listings（sellerListingSearch・読み取り）
  3. サンプルコードを実装
    • 最小限のコードでAPI呼び出しを実装
    • エラーハンドリングは後回し
  4. 動作確認
    • 期待通りのレスポンスが返るか確認
  5. 次のAPIへ
    • 1つずつ段階的に移行

検証範囲の注意（重要）：MCP Serverは本番では読み取り専用（GET / GraphQL Query のみ）。createListing / updateListing / markOrdersPaid などの書き込み系はMCPでは実行できません。書き込み系の検証は次の2通りで行ってください：

  • 移行コードの生成はSkillsに依頼（例:「How do I migrate AddFixedPriceItem to modern eBay APIs?」）
  • 実出品せずの動作確認は、Listing APIの VALIDATE モード（createListing / relistListing / createSecondChanceListing の options.operationMode=VALIDATE）を使用。旧APIの VerifyAddFixedPriceItem / VerifyAddItem / VerifyRelistItem 等に相当し、実データを作らずに入力検証できます。

4.4.2 サンプルコード例

GraphQL API（Listing API）の検証

GraphQL（新 Listing API 等）は、固定のサンプルを載せる代わりに MCP / Skills で検証してください（本番は読み取りのみ）。

  • MCP に自然言語で読み取り依頼。例: List my active eBay seller listings（sellerListingSearch 相当）。正確なクエリ・取得フィールドは MCP / Skills が最新スキーマから生成・実行します。
  • GraphQL エンドポイントは https://graphqlapi.ebay.com/graphql（MCP が内部で使う口と同一）。marketplace は 入力フィールドで渡します（REST の X-EBAY-C-MARKETPLACE-ID ヘッダは Listing API では使いません）。

REST API（Fulfillment API ＝ getOrders）の呼び出し例（Python）

import requests

# 注意: 注文取得のRESTは Fulfillment API。GraphQL の Order API とは別物。
FULFILLMENT_API_ENDPOINT = "https://api.ebay.com/sell/fulfillment/v1/order"
ACCESS_TOKEN = "your_access_token"

headers = {
    "Authorization": f"Bearer {ACCESS_TOKEN}",
    "Content-Type": "application/json"
}

# 注文一覧を取得
response = requests.get(
    f"{FULFILLMENT_API_ENDPOINT}?limit=10",
    headers=headers
)

print(response.json())

5. FAQ

5.1 ツール関連

Q1: MCPが動かない（"Server not found"エラー）

A: 以下を確認してください：

  3. 絶対パスの確認（ファイルが存在するか）

Q2: Skillsが起動しない

  1. インストール確認 
    npx skills list
  2. 配置場所の確認 
    ls ~/.config/skills/  # または適切なディレクトリ
  3. AIアシスタントの再起動

A: 以下を確認してください：

  2. スコープの確認
    • 必要なスコープが付与されているか確認
  3. Production vs Sandbox
    • MCPは現在Productionのみサポート

5.2 API移行関連

Q4: 移行先APIが見つからない

A: 以下の手順で確認してください：

  1. Claude Code + Skillsで質問 How do I migrate [API名] to modern eBay APIs?

  2. それでも見つからない場合は、Tech Supportにエスカレーション

  3. ebayjapan-techsupport@ebay.com

Q5: 1対1の対応がないAPIはどうすればいい？

A: Modern APIsでは機能が統合・分割されているため、複数のAPIを組み合わせる必要がある場合があります。

例:

  • CompleteSale → 用途別に5つのAPIへ分割（支払い: markItemCommitmentsPaid / markOrdersPaid、配送: createShippingFulfillment、評価: leaveFeedback、INR: provideShipmentInfoForItemNotReceivedInquiry）
  • GetOrders → 支払済（Order API + Fulfillment API）と未払い（Item Commitment API）で参照先が分かれる

詳細はSkillsで確認するか、Tech Supportにお問い合わせください。

Q6: Sandbox環境で検証したい

A: 現在、提供しているMCPはProductionのみ対応しています。

Sandbox環境での検証は、以下の方法で可能です：

  1. eBay Developer Program でSandbox環境にアクセス
  2. Postman等のAPIクライアントツールで手動検証
  3. 独自にSDKを使って実装

Q7: OAuth 2.0の実装が難しい

A: 以下のリソースをご参照ください：

  • OAuth 2.0 Tutorial
  • SDK一覧
  • Java, Python, Node.js, PHP等のSDKがOAuth実装をサポート

実装サンプルが必要な場合は、Tech Supportにご連絡ください。

GA（General Availability） 一般提供。誰でも本番利用できる正式リリース段階（beta・限定公開・Pilot より後）。本ガイドの「既存Modern」はこの状態。
